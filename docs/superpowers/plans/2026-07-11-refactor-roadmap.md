Реализация спеки `docs/superpowers/specs/2026-07-11-monolith-refactor-design.md`.
Подход C, инкрементально, поведение поверхностей строго 1:1.

Каждая фаза = отдельный план, дающий рабочий тестируемый результат. Планы Фаз 2–5
детализируются точечно перед выполнением, т.к. их шаги зависят от результата Фазы 1
(имена сервисов, сигнатуры). Детализировать раньше — значит писать догадки (= плейсхолдеры).

### Фаза 4 — что сделано
`GigaTranscriberQtApp` разбит на mixin-модули (класс сохраняет идентичность → GUI-тесты и entrypoint не меняются):
- `src/gui/llm_mixin.py` — `LlmMixin`: обработка LLM (диспетч, сохранение, экспорт, компактизация ошибок).
- `src/gui/llm_ui_mixin.py` — `LlmUiMixin`: построение вкладки LLM (`_create_llm_*`, диалог настроек).
- `src/gui/download_mixin.py` — `DownloadMixin`: загрузка медиа по ссылке.
- `src/gui/theme_mixin.py` — `ThemeMixin`: `_apply_theme` (палитра + QSS, ~390 строк).
Новые тесты: `tests/test_gui_llm_mixin.py`. Source-scrape тесты темы обновлены на новый модуль. Остаток app_qt (~2616) — ядро окна (конструирование, файлы, оркестрация обработки, прогресс) — сознательно оставлено монолитным как связная сердцевина.

**Гейт между фазами:** `ruff check` изменённых файлов + `pytest` зелёный (не хуже baseline: 2 pre-existing GUI-падения) + smoke изменённых поверхностей.

## Решение по Фазе 3 (сужение объёма — обосновано)

Полный router/package-split `api.py`/`web_app.py` **сознательно не выполняется**, потому что он нарушил бы строгое 1:1 и контракт тестов:

1. **Entrypoints.** `api.py` запускается как `python api.py` (README), `web_app` — как `web.web_app:app` (Dockerfile). Превращение `api.py` в пакет `api/` ломает `python api.py` — это задокументированное поведение (нарушение 1:1).
2. **Контракт тестов.** Тесты патчат 30+ атрибутов модулей: `api.VALID_API_KEY_HASHES`, `api.verify_api_key`, `api._hash_key`, `web_app.tasks_storage` (18×), `web_app.log_queues`, `web_app._restore_*`, `web_app.save_json_atomic` и т.д. `verify_api_key` обязан читать патчимый `api.VALID_API_KEY_HASHES` из namespace `api`; restore-функции держат общий мутабельный реестр задач. Вынос в подмодули сломал бы патчи без переписывания как surface-кода, так и тестов — при реальном риске регресса auth/persistence.
3. **Цель фазы уже достигнута.** Задача Фазы 3 — «тонкие роуты над сервис-слоем». После Фазы 1 роуты `api.py`/`web_app.py` уже делегируют валидацию, health, LLM, схему задач и сборку процессора в `src/services/`. Дальнейшее дробление — движение ради движения (риск option C, который мы договорились не брать).

Итог: backend декомпозирован настолько, насколько безопасно при строгом 1:1. Усилие декомпозиции перенесено в Фазу 4 (GUI), где оно безопасно (класс сохраняет идентичность через mixin-и) и высокоценно.

## Обнаруженные дивергенции (зафиксировать тестами, не «чинить» молча)

Обнаружены при чтении кода на этапе планирования; каждая — риск нарушить 1:1 при наивном
объединении дублей:

1. **`is_supported_format`** — `api.py` использует `SUPPORTED_FORMATS[1].split()` со срезом `*`;
   `web_app.py` — членство в `MEDIA_EXTENSIONS`. Разные источники допустимых расширений.
   → `file_policy` параметризуется источником; каждая поверхность сохраняет свой.
2. **LLM пустой ответ** — GUI (`app_qt.py`) бросает «… вернул пустой ответ» для Claude Code/Codex;
   `web_app.py` не проверяет (возвращает пустую строку). → сервис поддерживает флаг
   `strict_empty`; GUI=True, web=False (сохранение 1:1). Расхождение помечено как латентный
   баг web — вынести пользователю отдельно, не менять в этом рефакторинге.
3. **Провайдер «Other/Другое»** — GUI сверяет `_normalize_llm_provider(...)=="Other"`;
   web — `provider=="Другое"`. → нормализацию имени провайдера инкапсулировать в сервисе,
   адаптеры передают своё сырое значение.
4. **Текст ошибки неизвестного провайдера** различается (GUI локализует через `_t`).
   → сервис бросает нейтральный ключ/исключение; текст формирует адаптер.
