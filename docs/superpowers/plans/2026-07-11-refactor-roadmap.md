# Рефакторинг монолитов — дорожная карта планов

Реализация спеки `docs/superpowers/specs/2026-07-11-monolith-refactor-design.md`.
Подход C, инкрементально, поведение поверхностей строго 1:1.

Каждая фаза = отдельный план, дающий рабочий тестируемый результат. Планы Фаз 2–5
детализируются точечно перед выполнением, т.к. их шаги зависят от результата Фазы 1
(имена сервисов, сигнатуры). Детализировать раньше — значит писать догадки (= плейсхолдеры).

| Фаза | План | Статус | Риск | Итог фазы |
|---|---|---|---|---|
| 0 | `2026-07-11-phase-0-repo-hygiene.md` | ✅ написан | низкий | чистое дерево, зелёный baseline |
| 1 | `2026-07-11-phase-1-service-layer.md` | ✅ написан | низк-средн | `src/services/` без дублей; монолиты делегируют |
| 2 | `phase-2-core-formatters.md` | ⏳ по факту | низкий | форматтеры вынесены из `processor.py` |
| 3 | `phase-3-backend-routers.md` | ⏳ по факту | средний | `api/` и `web/` — тонкие роутеры |
| 4 | `phase-4-gui-decomposition.md` | ⏳ по факту | высокий | `app_qt.py` → пакет `src/gui/` |
| 5 | `phase-5-cli-and-cleanup.md` | ⏳ по факту | низк-средн | `cli/`, удалены обёртки-делегаты |

**Гейт между фазами:** `ruff check .` + `pytest` зелёные + smoke изменённых поверхностей.

## Обнаруженные дивергенции (зафиксировать тестами, не «чинить» молча)

Обнаружены при чтении кода на этапе планирования; каждая — риск нарушить 1:1 при наивном
объединении дублей:

1. **`is_supported_format`** — `api.py` использует `SUPPORTED_FORMATS[1].split()` со срезом `*`;
   `web_app.py` — членство в `MEDIA_EXTENSIONS`. Разные источники допустимых расширений.
   → `file_policy` параметризуется источником; каждая поверхность сохраняет свой.
2. **LLM пустой ответ** — GUI (`app_qt.py`) бросает «… вернул пустой ответ» для Claude Code/Codex;
   `web_app.py` не проверяет (возвращает пустую строку). → сервис поддерживает флаг
   `strict_empty`; GUI=True, web=False (сохранение 1:1). Расхождение помечено как латентный
   баг web — вынести пользователю отдельно, не менять в этом рефакторинге.
3. **Провайдер «Other/Другое»** — GUI сверяет `_normalize_llm_provider(...)=="Other"`;
   web — `provider=="Другое"`. → нормализацию имени провайдера инкапсулировать в сервисе,
   адаптеры передают своё сырое значение.
4. **Текст ошибки неизвестного провайдера** различается (GUI локализует через `_t`).
   → сервис бросает нейтральный ключ/исключение; текст формирует адаптер.
