> Полный технический аудит проекта (~5 900 LOC, 23 Python-модуля): FastAPI-сервер (`api.py`), CLI (`cli.py`), PyQt6 GUI (`src/gui/app_qt.py`), ядро транскрибации (`src/core/`), утилиты (`src/utils/`).
> Дата аудита: 2026-06-17.

Каждый пункт содержит: серьёзность, ссылку `файл:строка`, проблему и конкретное исправление. Работать сверху вниз — фазы упорядочены по убыванию риска.

## Фаза 0 — Критично: безопасность и потеря данных ✅ ВЫПОЛНЕНО (2026-06-17)

> Все пункты 0.1–0.9 реализованы и покрыты тестами (`tests/test_api_security.py`,
> `tests/test_api_integration.py`, `tests/test_atomic_json.py`, `tests/test_processing_stats.py`).
> Прогон: 42 passed в conda-окружении `gigaam`. Добавлен общий хелпер
> `src/utils/atomic_json.py`; дублирование загрузки в API вынесено в `_save_upload`/`_register_task`.

### 0.1 — CORS: `*` + credentials  🔴
**`api.py:572-574`** — `allow_origins=["*"]` вместе с `allow_credentials=True`. Невалидная и небезопасная комбинация: разрешает любому сайту делать запросы к API.
**Fix:** вынести список доменов в env (`CORS_ORIGINS.split(",")`); поскольку аутентификация по заголовку `X-API-Key`, поставить `allow_credentials=False`; сузить `allow_methods`/`allow_headers`.

### 0.2 — Все 4xx маскируются под 500  🔴
**`api.py:705, 708` и `api.py:829, 832`** — `raise HTTPException(413/400)` находится внутри `try`, который ловится `except Exception` и пере-оборачивается в 500. Клиент никогда не видит корректный код «файл слишком большой»/«неверный формат».
**Fix:** добавить `except HTTPException: raise` **перед** `except Exception` в обоих обработчиках загрузки.

### 0.3 — Path traversal через `file.filename` и `task_id`  🔴
**`api.py:666, 790`** — `file.filename` (контролируется клиентом) подставляется в путь напрямую (`../../...` выходит за `UPLOAD_DIR`).
**`api.py:1024, 1132, 1233, 1327-1333, 1400-1407`** — `task_id` из URL используется для `RESULTS_DIR / task_id` и деструктивного `shutil.rmtree`, без проверки формата.
**Fix:** санитизировать имя через `Path(file.filename).name`; хранить под `UPLOAD_DIR / task_id / safe_name`; валидировать `task_id` регуляркой `^[0-9a-f]{32}$` в зависимости/path-параметре до любой работы с ФС.

### 0.4 — API-ключи в плейнтексте + сравнение не constant-time  🔴
**`api.py:153-172, 177`** — ключи хранятся в `.api_keys` открытым текстом, первый ключ печатается в stdout; проверка `x_api_key not in VALID_API_KEYS` не constant-time.
**Fix:** хранить только SHA-256 хэши; сравнивать через `hmac.compare_digest`; не печатать сырой ключ (писать только в защищённый файл).

### 0.5 — `cleanup_old_tasks` объявлена, но не запущена → утечка диска/памяти  🔴
**`api.py:246` (объявление), `api.py:503-564` (lifespan без планировщика)** — старые задачи и файлы накапливаются вечно (`tasks_storage` растёт неограниченно → деградация и DoS со временем).
**Fix:** в `lifespan` запустить `asyncio.create_task(_cleanup_loop())`, отменять на shutdown.

### 0.6 — Утечка трейсбеков и листингов директорий в ответах API  🟠→🔴
**`api.py:471, 1014, 1117, 1182-1185, 1276-1279`** — полный Python-трейсбек кладётся в `tasks_storage` и отдаётся через `TaskStatus.error_details` (`api.py:105`); ошибки скачивания возвращают листинг папки и общее число задач.
**Fix:** отдавать клиенту обобщённые сообщения; трейсбеки только в логи; убрать `error_details` из модели ответа или закрыть за debug-флагом.

### 0.7 — Нет лимита размера до записи на диск (DoS)  🟠→🔴
**`api.py:670-683, 794-807`** — `MAX_FILE_SIZE` по умолчанию 2 ГБ, проверяется только в процессе записи; batch принимает до 10×2 ГБ; нет проверки `Content-Length` и общей квоты.
**Fix:** ранний отказ по `request.headers["content-length"]`; снизить дефолт; добавить глобальную/пер-аккаунт квоту диска; проверка `content-type` загрузки.

### 0.8 — Атомарность записи JSON (защита истории/настроек от порчи)  🟠→🔴
**`processing_stats.py:32`, `user_settings.py:38`** — `json.dump` в режиме `'w'` (сначала truncate). Падение/kill/конкурентная запись посреди дампа оставляет битый JSON → вся история тихо теряется при следующей загрузке.
**Fix:** писать во временный файл в той же папке + `os.replace(tmp, target)` (атомарно на POSIX/Windows). Вынести в общий хелпер `JsonStore`.

### 1.1 — Гонки в `tasks_storage` (нет блокировки)  🟠
**`api.py:85` (объявление), мутации в `331, 353, 445, 467, 1336, 1410`, итерация+удаление в `cleanup_old_tasks:252` и `delete_all_tasks:1380/1410`** — конкурентные фоновые обновления и удаление при итерации → `RuntimeError: dictionary changed size` / воскрешение удалённых задач.
**Fix:** `asyncio.Lock` вокруг операций; итерация по `list(...)`-снимку; проверять `if task_id in tasks_storage` перед апдейтом в горячем пути (`353, 445, 467`).

### 1.2 — Glob-хак выбора входного файла берёт неправильный/недописанный файл  🟠
**`api.py:283-407` (особенно `314-326, 390-392`)** — 120 строк ретраев + `glob(f"*{task_id}*")[0]`, используется файл, который «может ещё писаться». Причина-маскировка: файл уже `fsync`-нут и проверен до планирования задачи (`687-700`).
**Fix:** удалить всю glob/retry-машинерию; доверять переданному `file_path`; при отсутствии — чисто падать.

### 1.3 — GUI: общее состояние читается/пишется из worker-потока  🟠
**`src/gui/app_qt.py:1004, 1050, 1052`** — `_process_files` (raw `threading.Thread`) читает `enable_diarization`, `output_formats`/чекбоксы и пишет `files_processed`/`time_spent`, которые одновременно читает `QTimer`-слот на GUI-потоке. `num_speakers` уже корректно снимается на главном потоке (`987`) — сделать так же для остального.
**Fix:** снять `enable_diarization` и список форматов на GUI-потоке и передать аргументами в `_process_files`; блокировать чекбоксы форматов/диаризации на время обработки.

### 1.4 — GUI: «ОЧИСТИТЬ ВСЕ»/закрытие разрушают состояние работающего worker'а  🟠
**`src/gui/app_qt.py:874-896, 1111, 1166-1174`** — `_clear_all` обнуляет `total_files`/`files_to_process`/`start_time`, пока worker индексирует `files_to_process[files_processed]` и считает `time()-start_time` → `IndexError`/`ZeroDivisionError`/`TypeError`. `closeEvent` всегда `accept()`, осиротляя daemon-потоки.
**Fix:** не мутировать общие коллекции при живом worker; флаг отмены, который worker проверяет; сброс состояния только в `_on_processing_finished`; в `closeEvent` при активной работе — подтверждение + запрос отмены.
